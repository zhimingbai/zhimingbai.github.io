文件状态​

git中的文件有两种状态：未跟踪和已跟踪。未跟踪指文件没有被git所管理，已跟踪指文件已被git管理。已跟踪的文件又有三种状态：未修改、修改和暂存。

暂存，表示文件修改已经保存，但是尚未提交到git仓库。

未修改，表示磁盘中的文件和git仓库中文件相同，没有修改。

已修改，表示磁盘中文件已被修改，和git仓库中文件不同。

可以通过git status来查看文件的状态

分支​

git在存储文件时，每一次代码的提交都会创建一个与之对应的节点，git就是通过一个一个的节点来记录代码的状态的。节点会构成一个树状结构，树状结构就意味着这个树会存在分支，默认情况下仓库只会有一个分支，命名为master。在使用git的时候，可以创建多个分支，每个分支之间相互独立，在一个分支上修改的代码不会影响到其他的分支。

git branch # 查看当前分支
git branch <branch name> # 创建新的分支
git branch -d <branch name> # 删除分支
git switch <branch name> # 切换分支
 git switch -c <branch name> # 创建并切换分支

在开发中，都是在自己的分支上编写代码，代码编写完成后，再讲自己的分支合并到主分支中。

变基​

在实际的开发过程中，我们除了可以通过==merg==来合并分支之外，还可以通过进行==变基==来完成分支的合并。

我们通过merge合并分支时，在提交记录中会将所有的分支创建和分支合并的过程全部都显示出来，这样当项目比较复杂，开发过程比较波折时，我必须要反复的创建、合并、删除分支。这样一来将会使得我们代码的提交记录变得极为混乱。

在Git中，“变基”（Rebase）是一种操作，用于将一个分支上的提交应用到另一个分支的顶部。变基的目的是让一个分支的历史看起来像是基于另一个分支的最新状态发展起来的，从而保持线性的提交历史。

变基的基本原理：

  1. 创建新的提交：

    • 当你对一个分支进行变基时，Git会创建一系列新的提交来反映原分支的变更。这些新的提交不是简单地复制原提交，而是根据原提交与目标分支之间的差异来创建的。

  2. 移动分支指针：

    • 变基完成后，原分支的指针（通常是HEAD）会被移动到新创建的最后一个提交上。

  3. 解决冲突：

    • 如果变基过程中涉及到的提交与目标分支之间存在冲突，Git会停止变基过程并提示你解决这些冲突。一旦冲突解决，你可以继续变基过程。

变基的基本流程：

假设你有两个分支：main 和 feature，feature 是基于 main 的一个较早版本创建的。下面是如何使用 git rebase 来更新 feature 分支以基于 main 的最新状态：

  1. 切换到需要变基的分支：

    git checkout feature

  2. 开始变基过程：

    git rebase main

    这会将 feature 分支上的提交应用到 main 分支的最新提交之上。

  3. 解决冲突（如果有的话）：

    • 如果在变基过程中出现了冲突，Git会停止变基过程并让你解决这些冲突。解决冲突后，你可以继续变基： 
      git add <conflicted-file>
      git rebase --continue

  4. 完成变基：

    • 如果没有冲突，或者所有冲突都已解决，变基过程将自动完成，feature 分支的指针会移动到新提交的顶部。

变基的优点：

  • 清晰的历史：变基可以使提交历史更加线性，更容易阅读和理解。
  • 易于集成：变基后的分支更容易合并到其他分支中，因为它们看起来就像是在目标分支的基础上直接构建的。

变基的注意事项：

  • 变基会改变提交历史：变基会创建新的提交，这可能会改变原有的提交历史。对于公共分支（即已经被其他人获取的分支），谨慎使用变基，以免引起混乱。
  • 避免变基已推送到共享仓库的提交：如果你的分支已经被其他人拉取或合并，那么不应该对该分支进行变基，否则会引发历史记录的不一致。

如果你有任何具体的问题或需要进一步的帮助，请随时提问。

远程仓库​

目前我对于 git 所有操作都是在本地进行的。在开发中显然不能这样的，这时我们就需要一个远程的 git 仓库。远程的 git 仓库和本地的本质没有什么区别，不同点在于远程的仓库可以被多人同时访问使用，方便我们协同开发。在实际工作中，git 的服务器通常由公司搭建内部使用或是购买一些公共的私有 git 服务器。我们学习阶段，直接使用一些开放的公共 git 仓库。目前我们常用的库有两个：GitHub 和 Gitee（码云）

将本地库上传 github：

git remote add origin https://github.com/zhimingbai/git-demo
# git remote add <remote name> <url>

git branch -M main
# 修改分支的名字的为main

git push -u origin main
# git push 将代码上传服务器上

将本地库上传 gitee：

git remote add gitee https://gitee.com/angelica-tea/git-study.git
git push -u gitee main

  1. 远程库的操作的命令

    git remote # 列出当前的关联的远程库
    git remote add <远程库名> <url> # 关联远程仓库
    git remote remove <远程库名>  # 删除远程库
    git push -u <远程库名> <分支名> # 向远程库推送代码，并和当前分支关联
    git push <远程库> <本地分支>:<远程分支>
    git clone <url> # 从远程库下载代码

    git push # 如果本地的版本低于远程库，push默认是推不上去
    git fetch # 要想推送成功，必须先确保本地库和远程库的版本一致，fetch它会从远程仓库下载所有代码，但是它不会将代码和当前分支自动合并
    		 # 使用fetch拉取代码后，必须要手动对代码进行合并
    git pull  # 从服务器上拉取代码并自动合并

    注意：推送代码之前，一定要先从远程库中拉取最新的代码
